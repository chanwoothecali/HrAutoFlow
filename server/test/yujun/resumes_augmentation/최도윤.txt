[경력]

로그플로우(LogFlow) / 백엔드 & 플랫폼 엔지니어
2021.03 ~ 현재 (3년 9개월, 정규직, 용인/서울 혼합 근무)

1) 로그/이벤트 수집 파이프라인 백엔드 개발 및 운영
- 기술: Go, gRPC, Kafka, Redis, PostgreSQL, Kubernetes(EKS), Prometheus, Grafana
- 역할:
  - 웹/모바일 클라이언트에서 발생하는 행동 로그/이벤트 수집용 Ingestion API 및 gRPC 엔드포인트 설계·구현
  - Kafka 토픽 설계(파티션 전략, 키 설계) 및 Consumer 그룹 스케일링 전략 수립
  - Redis 기반 Rate Limiting, 간단한 큐 버퍼링으로 피크 타임 트래픽 완충 로직 구현
  - 수집 파이프라인 장애 시 재처리/중복 처리 방지를 위한 idempotent 키 설계
- 성과:
  - 분당 처리 가능한 이벤트 수를 약 3만 건 → 12만 건으로 확장 (클러스터 스케일링 및 Consumer 구조 개선)
  - 피크 타임 10분 동안 발생하던 메시지 유실 사례를 0건으로 감소 (재시도/Dead Letter Queue 도입 후)
  - 주요 Ingestion API P95 응답 시간 350ms → 150ms 수준으로 개선

2) 내부 지표 조회용 Aggregation 서비스 개발
- 기술: Kotlin, Spring Boot, PostgreSQL, MongoDB, Redis, gRPC, REST
- 역할:
  - 마케터/PO가 사용하는 대시보드를 위한 지표 Aggregation 서비스 설계·구현 (활성 사용자 수, 전환율, 퍼널 분석 등)
  - 비동기 집계 배치 및 사전 집계 테이블 구조 설계로, 클라이언트 쿼리 응답 속도 최적화
  - gRPC 기반으로 다른 서비스와 연동되는 공용 지표 조회 API 정의
- 성과:
  - 대시보드 주요 지표 조회 응답 시간 평균 2.5초 → 400ms로 단축
  - 사내 다른 팀(마케팅, CRM)에서 동일 Aggregation API를 재사용하면서, 중복 구현되던 통계 로직 3개 이상 제거

3) 공용 Observability 플랫폼 고도화
- 기술: Prometheus, Grafana, Loki, OpenTelemetry, Slack Webhook
- 역할:
  - 서비스 공통 지표/로그/트레이스를 한 번에 확인할 수 있도록 대시보드 템플릿 설계
  - gRPC/HTTP 서비스 공용 미들웨어에 Trace ID, Latency 로그 자동 삽입 로직 추가
  - 알람 룰(장애 레벨, 지연 증가, 에러율 상승 등)을 표준화하고 팀별로 재사용 가능한 룰셋 정의
- 성과:
  - 장애 발생 시 원인 서비스 식별까지 걸리던 평균 시간 약 30% 단축 (내부 회고 기준)
  - 신규 마이크로서비스가 Observability 템플릿을 그대로 적용해, 대시보드/알람 셋업 시간을 반나절 이내로 단축

[자기 소개 및 강점 요약]
- 여러 서비스가 엮인 환경에서 "어디서 문제가 시작됐는지"를 빠르게 찾을 수 있는 관측 가능성(Observability)에 관심이 많습니다.

- 단일 서비스 개선에 그치지 않고, 템플릿/플랫폼화 작업을 통해 팀 전체 생산성과 품질을 높이는 일을 좋아합니다.
- 새로운 기술을 도입할 때는, 작은 PoC로 검증하고 팀원들과 공유·논의한 후 점진적으로 확대 적용하는 편입니다.
